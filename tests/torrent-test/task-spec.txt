# Torrent Testing Deep Dive

## Objective
Deeply test torrent creation, seeding, and downloading with multiple simultaneous clients.

## Test Data (already copied to source/)
15 files, 485KB total from 3 Gutenberg books:
- q-science/38755/ (~157KB, .mobi .epub .txt .html.zip .info)
- h-social-sciences/62157/ (~164KB, same file types)
- d-world-history/62152/ (~165KB, same file types)

## Directory Structure
/test/torrent-test/
├── source/          (test data with preserved directory structure)
├── seeder/          (creates and seeds the torrent)
├── peer1/           (downloads torrent client 1)
├── peer2/           (downloads torrent client 2)
├── peer3/           (downloads torrent client 3)
└── results/         (test output and analysis)

## Requirements

### 1. Torrent Creation
- Copy test data to source/ preserving directory structure
- Create torrent file from source/
- Start seeder in seeder/ directory
- Record magnet link

### 2. Multi-Peer Testing
- Run 3 peer clients simultaneously
- All download from same magnet link
- Track: download speeds, completion times, connections

### 3. Data Integrity
- Verify all peers have identical files
- SHA256 checksums across original + peers
- File count and size verification

### 4. Search Testing
- Search across downloaded content
- Find specific text patterns in .txt files
- Test binary file search (in .epub/.mobi)
- Compare search results between peers

### 5. Report Results
- Performance metrics
- Integrity verification results
- Search test results
- Observations and issues

## Status

**Automated — fixture is fully operational.**

The scenario is driven end-to-end by `scripts/test-multi-peer.sh`:

- `scripts/gen-test-fixture.sh` generates the deterministic 15-file, ~310 KiB corpus
  under `tests/torrent-test/source/` (3 books × 5 file types with searchable text markers).
- `scripts/test-multi-peer.sh` orchestrates 4 real `dist/swartznet` binary processes on
  loopback (1 seeder + 3 peers), each with its own isolated data/index directory and port.
  Peer discovery uses the `x.pe=` magnet URI parameter so no DHT or tracker is required.
  The script asserts:
  1. Seeder verifies source tree (hash check complete before serving).
  2. All 3 peers download 100% of the content.
  3. Per-file SHA-256 integrity against the original source tree.
  4. Each peer's Bleve index answers a text-search for the corpus keyword "photon".
  Timestamped run reports are written to `tests/torrent-test/results/run-<timestamp>.txt`.

First passing run: 2026-04-20. All 3 peers downloaded 316 KiB in under 2 s on loopback;
search found 48 hits per peer once the ingest pipeline completed.
